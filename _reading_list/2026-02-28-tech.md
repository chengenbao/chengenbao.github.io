## 1. Mixture of Experts (MoEs) in Transformers — 工程实现全景

**来源：** [Hugging Face Blog](https://huggingface.co/blog/moe-transformers) · 2026-02-26

### 核心摘要
MoE 模型在稠密 Transformer 的 FFN 层引入专家池，每个 token 由路由器选择少量专家处理，使模型容量与推理计算解耦。以 gpt-oss-20b 为例：21B 总参数但仅激活 ~3.6B 参数/token，在 M3 Ultra 上实测 ~115 token/s，与理论预估高度吻合。DeepSeek R1 的爆火（2025年1月）已成为 MoE 大规模落地的明确拐点，Qwen 3.5、MiniMax M2、Kimi K2.5 等相继跟进。本文聚焦 Transformers 库对 MoE 的工程适配，涵盖加载、量化、设备放置到推理后端的全链路改造。

### 技术要点
- **稀疏激活机制**：Router 依据 token 隐层表示动态选 top-k 专家，推理速度近似等效于激活参数量的稠密模型
- **专家并行（Expert Parallelism）**：Expert 构成天然并行边界，不同 token 路由至不同 GPU 的专家，大幅降低通信依赖
- **Transformers 库适配挑战**：原有加载/量化/推理路径均为稠密模型设计，MoE 需改造 device_map、量化粒度、KV cache 管理
- **行业采纳趋势**：闭源（ChatGPT）与开源（DeepSeek、Mixtral、Llama 4-Scout）均大规模使用 MoE 架构

### 为什么值得读
MoE 已成为 2025-2026 大模型默认架构选型。这篇文章系统梳理了从理论到 Transformers 库工程实现的全流程，是理解 MoE 落地工程复杂度的最佳入门材料，配有完整代码路径说明。

## 2. Supercharging LLMs: Scalable RL with torchforge and Weaver

**来源：** [PyTorch Blog](https://pytorch.org/blog/supercharging-llms-scalable-rl-with-torchforge-and-weaver/) · 2026-01-09

### 核心摘要
Meta PyTorch 团队开源 **torchforge**，一个 PyTorch 原生的大规模 RL post-training 框架，通过与 Stanford、CoreWeave 合作，在 512 GPU 集群上跑通 GRPO 训练。Forge 的核心价值在于将 RL 算法与分布式基础设施解耦：研究者用伪代码式 API 描述算法，Forge 自动处理 GPU 资源调度、权重同步和故障恢复。配合 Weaver（弱验证器集成系统），可在无需人工标注的情况下提供生产级 reward 信号。在 Qwen3-8B/32B 上的数学、GPQA、MMLU Pro 基准测试中，Weaver 方案接近标注数据上界。

### 技术要点
- **Forge 三层架构**：Monarch（分布式协调+容错）、TorchStore（RDMA 加速权重同步）、vLLM（高吞吐推理）
- **解耦控制面与数据面**：Monarch 负责路由/生命周期，TorchStore 做 DTensor 原生张量移动，GPU 生成与权重同步并行进行
- **灵活同步性**：支持从全同步 PPO 到完全异步 off-policy，同一代码路径无缝切换
- **Weaver 弱验证器**：集成多个验证器 ensemble，解决 LLM 生成与验证之间的能力不对称问题
- **RLVR（带验证器奖励的强化学习）**：实现无标注数据的高质量 reasoning 提升

### 为什么值得读
torchforge 首次让中小团队具备在百 GPU 规模上进行 RL post-training 的基础设施，且完全开源。对于研究 RLHF/RLAIF 或 reasoning 训练的工程师，这是目前最生产就绪的 PyTorch 原生方案。

## 3. Efficient MoE Pre-training at Scale on 1K AMD GPUs with TorchTitan

**来源：** [PyTorch Blog](https://pytorch.org/blog/efficient-moe-pre-training-at-scale-with-torchtitan/) · 2025-12-01

### 核心摘要
AMD 与 Meta PyTorch 团队合作，在 1,024 张 MI325X GPU 上对 TorchTitan 和 Primus-Turbo（AMD 开源内核库）进行深度调优，实现了 MoE 预训练的近理想扩展性。DeepSeek-V3-671B 获得 2.77× 速度提升，从 128 到 1,024 GPU 的扩展效率达 96%；Llama 4-Scout 在 32→512 GPU 范围内实现完美线性扩展。核心挑战包括：专家 micro-GEMM 的 GPU 利用率问题、All-to-All 通信瓶颈、以及多路并行策略（FSDP、流水线、Tensor、Expert Parallelism）的协同调度。

### 技术要点
- **Primus-Turbo FP8 内核**：AMD 专为 MoE token dispatch 设计的 FP8 混合精度内核，解决 micro-GEMM 效率问题
- **Expert Parallelism 集成**：在 TorchTitan 的统一并行框架中叠加 Expert Parallelism，仅修改 TOML 配置文件
- **96% 扩展效率**：通过计算-通信 overlap、异步 All-to-All，将 Expert Parallelism 的通信开销从瓶颈降为背景噪声
- **统一配置驱动**：同一代码路径覆盖 Dense（Llama 3）、Hybrid（Qwen 3）、Sparse MoE（DeepSeek-V3、Llama 4）

### 为什么值得读
稀疏 MoE 在 AMD GPU 上实现接近 NVIDIA 生态的训练效率，意味着 AI 算力供给不再受单一硬件制约。文章详细拆解了每个优化点，对需要在非 NVIDIA 硬件上部署 MoE 训练的团队极具参考价值。

## 4. Custom Kernels for All from Codex and Claude

**来源：** [Hugging Face Blog](https://huggingface.co/blog/custom-cuda-kernels-agent-skills) · 2026-02-13

### 核心摘要
HuggingFace 构建了一个"Agent Skill"，将 CUDA 内核开发的专业知识（硬件架构参数、内存访问模式、PyTorch binding 规范）打包成 Coding Agent 可加载的上下文，让 Claude 和 Codex 能端到端生成适用于 diffusers/transformers 的生产级 CUDA 内核。实测：Agent 成功为 Qwen3-8B 生成 RMSNorm 和 Attention 的向量化 H100 内核，并附带完整 benchmark。这与 HF Kernel Hub 形成闭环：Hub 负责分发编译好的内核，Skill 负责生成这些内核。

### 技术要点
- **Agent Skill 范式**：将领域知识打包为 Skill 文件（放入 `.claude/skills/` 或 `.codex/skills/`），Agent 按需加载，解决 CUDA 开发的"隐性知识"传递问题
- **硬件感知生成**：Skill 携带 H100/A100/T4 各代 GPU 的 shared memory 大小、compute capability、带宽参数，Agent 据此选择向量化策略和 warp shuffle 方案
- **PyTorch 集成规范**：自动生成 `torch.compile` 可识别的 binding，包含 `build.toml` 构建配置
- **一键安装**：`kernels skills add cuda-kernels --claude` 即完成多 Agent 平台（Claude Code、Cursor、Codex）的 Skill 安装

### 为什么值得读
CUDA 内核开发的专业门槛极高，这篇文章展示了 AI Agent 实质性降低这一门槛的可行路径。对于需要推理优化、自定义算子的团队，"AI 写 CUDA"从概念走向了可落地的工程实践。
